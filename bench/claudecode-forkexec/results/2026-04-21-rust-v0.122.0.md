# Claude Code fork/exec benchmark — `openai/codex@rust-v0.122.0`

* **Date**: 2026-04-21 (first Claude Code end-to-end run)
* **Claude Code CLI**: `2.1.116` (`claude -p --output-format stream-json`)
* **Target tree**: `openai/codex` at tag `rust-v0.122.0` (latest release)
* **Host**: macOS Darwin (arm64), single-sample N=1
* **Tracer**: `shim` (PATH-shadow wrappers, `$ZDOTDIR` override, `--add-dir` whitelist)
* **mcp-cli**: this repo at `main` post-`feat/bench-claudecode` +
  `fix/claudecode-bench-bare` + `fix/claudecode-bench-stdin`.
  Registered for this run via a throwaway `$OUT_DIR/mcp-config.json`
  passed as `--mcp-config` (no user-level `~/.claude.json` changes),
  with `--daemon-arg=--idle-timeout=30m` embedded so the daemon
  survives between the cold and warm passes.
* **Tool forcing**: cold and warm passes run with
  `--disallowed-tools "Bash,Read,Grep,Glob,Edit,Write,NotebookEdit"`,
  so the agent has to route all I/O and search through mcp-cli.
  Baseline has the full built-in palette.
* **Prompt**: [`bench/claudecode-forkexec/prompt.md`](../prompt.md)
  (symlinked to the codex twin — identical workload for apples-to-apples)
* **Companion**: [`bench/codex-forkexec/results/2026-04-21-rust-v0.122.0-sandbox-ablation.md`](../../codex-forkexec/results/2026-04-21-rust-v0.122.0-sandbox-ablation.md)
  — same prompt, same target, same host, same day, Codex instead
  of Claude.

## Headline

Cold mcp-cli cuts fork/exec **82 %** (85 → 15), beats baseline
wall-clock by **10 s** (208 s vs 219 s), and drops cold→warm cached
input by **952 k tokens** even though the warm agent makes *more*
calls — the in-memory `search_cache` + `parse_cache` + prewarm doing
exactly what they're supposed to.

Claude Code shells out more aggressively than Codex by default
(85 vs 62 execves on the same prompt), so the proportional fork/exec
win is bigger: **82 % here vs 66 % for Codex**. Claude also doesn't
regress wall-clock the way Codex did even in the sandboxed
measurement — the cold pass is actually *faster* than baseline.

## What each column proves

### baseline (vanilla claude, full built-ins)

85 execves across `cat`/`head`/`grep`/`find`/`ls`/`git`/`sed`/`awk` — heavier
shell-out behaviour than Codex's 64. The Bash + Grep built-ins
also fired once each, and this host has `context-mode` mounted
globally, so Claude additionally made 12 `ctx_*` MCP calls. 13 k
output tokens, ~220 s wall.

### cold mcp-cli (disallowed-tools + --mcp-config)

Disabling the built-in I/O + search palette forced Claude onto the
daemon. Outcome: **15 execves total** (the 2 `grep` are internal,
the 13 `git` are Claude's own startup — not part of the model's
tool palette), and 29 mcp-cli calls (17 × `fs_read`, 9 × `search_grep`,
3 × `fs_scan`). Notably, **zero context-mode fallback** — the
prompt's "prefer mcp-cli" nudge plus the disallowed list is enough
to keep Claude on the daemon.

Wall-clock: **208 s, 10 s *faster* than baseline**. Compact
structured returns from the daemon shrank the per-turn context the
model has to digest, so even with 29 atomic MCP calls Claude still
came out ahead.

### warm mcp-cli (same daemon, populated caches)

The daemon stayed alive (idle-timeout pinned at 30 min) so
`search_cache` + `parse_cache` are already populated and prewarm
has long since finished.

Claude chose to dig deeper now that every probe was cheap:
**53 mcp-cli calls** (27 × `fs_read`, 12 × `search_grep`,
**11 × `code_outline`** — Claude didn't use `code_outline` in the
cold pass, but reached for it in warm). This pushed wall-clock
up 88 s vs cold.

But the cache payoff is visible exactly where it should be:
**cached input drops 952 k tokens cold → warm** (1.95 M → 997 k)
even though the call count nearly doubled. Every `fs_read` that
hits `parse_cache`, every `search_grep` that hits the
ChangeLog-versioned `search_cache`, avoids the tree-sitter reparse
or grep re-walk.

## Cross-agent comparison (same prompt, same day)

|                               | Codex (no sandbox) | Claude Code       |
| ----------------------------- | -----------------: | ----------------: |
| execve baseline → cold        |     64 → 22 (−66%) | 85 → 15 (**−82%**) |
| wall-clock baseline → cold    |   209 → 117 (−44%) | 219 → 208 (**−5%**) |
| wall-clock cold → warm        |   117 → 159 (+36%) | 208 → 297 (+43%)   |
| cached input baseline → cold  |    1.81M → 572k (−68%) | 989k → 1.95M (+97%) |
| cached input cold → warm      |   572k → 1221k (+114%) | 1.95M → 997k (**−49%**) |

Two things jump out:

1. **Codex trims context tokens cold-vs-baseline; Claude Code
   doesn't** — Claude's cold pass *adds* cached input tokens
   (989 k → 1.95 M). Reason: Claude's built-in tools (`Bash`,
   `Read`, `Grep`) return compact CLI-style output, while our
   bench's baseline Claude also uses `ctx_*` tools which compact
   further. Once we strip those and route to mcp-cli, every file
   Claude reads is loaded into context (vs grepped-and-summarised
   via `ctx_execute`), so the raw input grows even as the agent
   does structurally less work.
2. **Claude's warm pass is where mcp-cli's caches earn their
   keep** — cached input drops 49 % cold → warm *despite more
   calls*. That's the prewarm + `parse_cache` + `search_cache`
   story, measured directly.

## What's actually proven

* **`--mcp-config` + `--disallowed-tools` is the right isolation
  pattern for Claude Code.** No pollution of the user's
  `~/.claude.json`; the agent actually routes through the daemon
  (zero built-in fallback observed).
* **mcp-cli delivers a Pareto improvement on Claude's cold pass**:
  fewer execves (15 vs 85), faster wall-clock (208 s vs 219 s),
  and the model actually did *less* gross work (11.7 k vs 13.6 k
  output tokens).
* **Warm pass confirms the in-memory-cache mechanism end-to-end**:
  cached input −952 k tokens cold → warm while call count rose
  from 29 to 53.

## Caveats

* **N = 1 per cell.** Model tool-choice variance is substantial
  (Claude's cold vs warm call counts and tool mix differ
  materially). Trust the direction and rough magnitude, not the
  third significant figure. Warm wall-clock regression is the
  cell most sensitive to re-running; the −82 % execve number
  should hold across runs.
* **`--bare` not used.** `--bare` blocks keychain reads, which
  breaks macOS OAuth — on this host there's no
  `ANTHROPIC_API_KEY` set explicitly, so `--bare` errors out with
  "Not logged in". The trade: the user's hooks, plugins, and
  `CLAUDE.md` auto-discovery run in all three passes. Bias is
  constant across baseline/cold/warm; comparative deltas stay
  meaningful, absolute numbers reflect the host.
* **Baseline had context-mode mounted.** This host has the
  `context-mode` MCP plugin globally registered, so Claude's
  baseline run used `ctx_batch_execute` / `ctx_execute` /
  `ctx_search` 12 times. These don't fork/exec (they're
  structured MCP returns), so execve comparisons are still clean,
  but token numbers bake in context-mode's own compaction. A
  fully-isolated Claude baseline (no global MCP servers) would
  likely show higher token counts in the baseline column and a
  bigger cold-vs-baseline win.

    MCP_CLI="$PWD/target/release/mcp-cli" \
    OUT_DIR=/tmp/mcpcli-claude-bench \
    bench/claudecode-forkexec/run.sh
