# Codex fork/exec benchmark — `openai/codex@rust-v0.121.0`, `search_grep ?context`

* **Date**: 2026-04-20 (fifth iteration)
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Config**: same as v2/v4 — `mcp-cli install --target codex --prefer-mcp`.
* **Change vs v4**: added a `context: u32` field to the `search_grep`
  bridge schema. Daemon uses `grep-searcher`'s native before/after
  context and a custom `ContextSink` to attach surrounding lines to
  each hit. No prose nudges anywhere.
* **Compare against**:
  [v2](./2026-04-20-rust-v0.121.0-prefer-mcp.md) (baseline prefer-mcp),
  [v3](./2026-04-20-rust-v0.121.0-batch-v1.md) (prose nudge regression),
  [v4](./2026-04-20-rust-v0.121.0-batch-v2.md) (trimmed prose).

## What actually moved

* **Codex reached for `context` spontaneously.** 31 of 32
  `search_grep` calls carried `context: 3`–`8`. The description
  update was just "set `context: N` to attach N lines before and
  after each match" — no "PREFER this", no imperative prose. The
  model inferred the tradeoff from the schema alone. (Contrast
  with v3, where prose nudges regressed both axes.)
* **`fs_read` count dropped from 36 → 21.** Most of the "read
  around this hit" follow-up calls that padded v2/v4 are gone;
  the context now arrives in the same grep response.
* **Total MCP calls went from 118 → 73.** 45 fewer atomic turns,
  each saving a full model-reasoning round-trip. Wall clock
  follows directly (404 s → 324 s, -20 %).

## What it cost

* **Input tokens up ~4 % vs v2** (1.76 M → 2.23 M) because the
  context lines have to be paid for somewhere. This is the
  right trade: 470 K extra input tokens buys back 80 s of wall
  clock on a single bench. Future cache-hit runs (same daemon
  across turns) amortise even further since the grep cache now
  stores the already-hydrated response.

## What this validates

* **Schema-level API design moves the model more reliably than
  prose in descriptions.** The v3 prose nudge ("PREFER
  fs_read_batch") got one use and a 2.5 M-token tax. The v5
  schema addition ("context: integer") got 31 uses for free.
* **Server-side turn-collapse beats agent-side composition.** We
  don't need codex to learn "grep then read around the hit" as a
  two-step sequence; we just need the result of the two-step
  sequence available in one tool call.

## What's next

The remaining wall-clock regression (+61 % vs bash baseline) is
the inherent cost of atomic MCP turns. Two candidates for the
next reduction, in order of leverage:

* **Batch `code_symbols` / `code_outline` by path list.** Not hit
  often enough in this single sample to be a clear win, but the
  access pattern is similar.
* **Keep the daemon warm across bench runs.** Each bench starts
  from zero cache; a second query to the same pattern today
  re-walks the tree. An LRU on the `search_grep` side (already
  exists!) just wasn't warmed because the bench is a single
  agent session.
