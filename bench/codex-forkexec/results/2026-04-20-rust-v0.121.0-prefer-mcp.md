# Codex fork/exec benchmark — `openai/codex@rust-v0.121.0`, `--prefer-mcp`

* **Date**: 2026-04-20 (re-run)
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Target tree**: `openai/codex` at tag `rust-v0.121.0` (latest stable)
* **Host**: macOS Darwin (arm64), single-sample N=1
* **Tracer**: `shim` (PATH-shadow wrappers, `$ZDOTDIR` override, `--add-dir` whitelist)
* **mcp-cli**: this repo at `feat/bash-rewrite-shim`, MCP plugin registered via
  `mcp-cli install --target codex --prefer-mcp` — the new flag writes
  `[features] shell_tool = false` and per-tool
  `approval_mode = "approve"` to the per-bench `$CODEX_HOME/config.toml`.
* **Prompt**: [`bench/codex-forkexec/prompt.md`](../prompt.md)
* **Compare against**: [v1 (no `--prefer-mcp`)](./2026-04-20-rust-v0.121.0.md)

## Headline finding

**Path 1 works.** With `--prefer-mcp`:

* **Fork/exec collapses 79 %** (103 → 22). Every agent-issued `rg`,
  `sed`, `head` invocation is gone; the residual 22 are codex's own
  startup, sandbox bootstrap, and `git` plumbing — totally unrelated
  to user tooling.
* **Input tokens drop 19 %** (-404 K). The daemon's responses are
  smaller per call than the equivalent bash output (compact mode on
  `git_status` / `search_grep` helps; even the raw forms are tighter
  than `rg` / `cat` text dumps).
* **124 MCP calls actually landed on the daemon** — `fs_read` ×50,
  `search_grep` ×70, `fs_scan` ×4. Compare to v1, where the same
  install (without `--prefer-mcp`) sent **zero** real RPCs and codex
  ran 88 bash commands.

## Wall clock regression (+45 %)

The flip side: codex took 117 s longer wall-clock with the daemon.
The reason is visible in the call counts — codex made
**~2.7× more atomic operations** (124 MCP calls vs. 45 bash
commands) because each MCP call is one tool, while a single bash
command can be a pipeline (`rg pattern | head -3 | sort`). More
turns → more model latency → longer wall.

This is a real ergonomics gap in the MCP toolset that the bench
surfaces cleanly:

* **No compound tools.** `search_grep | head` is two MCP turns
  today. A `search_grep_first_n` (or just better `max_results`
  hinting in the model context) would compress the agent's loop.
* **No batch tools.** Reading 50 files is 50 `fs_read` round-trips
  instead of one bash `for f in ...; do cat $f; done`. A
  `fs_read_batch(paths: Vec<String>)` would close the gap.
* **Per-call latency** is also worth profiling — if each `fs_read`
  is 50–100 ms (UDS round-trip + JSON), 50 of them adds up. The
  `metrics.tool_latency` snapshot is supposed to show this but
  didn't land in this run because the daemon idle-exited before the
  follow-up dump call hit it (known gap in `run.sh`).

## What's actually proven

* The `--prefer-mcp` install path closes the M5 v1 gap: the daemon
  is *used*, not just *mounted*.
* Per-call kernel overhead (the M5 thesis: fork/exec/dynamic linker
  per shell call) collapses to near zero for the binaries the
  daemon owns.
* Per-call agent-context overhead (input tokens) drops 19 %; the
  M7 compaction layer is doing real work even on this single
  sample.
* The remaining wall-clock cost is concentrated in agent-loop
  turns, not in the daemon. Closing it is an MCP-tool-shape
  problem (compound + batch tools, see above), not a daemon-perf
  problem.

## Reproduce

```sh
cargo build --release
PATH="$PWD/target/release:$PATH" \
    OUT_DIR=/tmp/mcpcli-codex-bench \
    bench/codex-forkexec/run.sh
```

The bench's `run.sh` now passes `--prefer-mcp` to
`mcp-cli install --target codex` and (separately fixed in this PR)
honours `$CODEX_HOME` so the per-bench config gets the flag's
effect — the previous `mcp-cli install` walked `$HOME/.codex/`
unconditionally, so the v2 attempt at this measurement quietly
mutated the user's real config and left the per-bench config
empty.
