# Codex fork/exec benchmark — `openai/codex@rust-v0.121.0`, with `fs_read_batch`

* **Date**: 2026-04-20 (third iteration, stacked on `feat/fs-read-batch`)
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Config**: `mcp-cli install --target codex --prefer-mcp` (as in v2)
* **Change vs v2**: new `fs_read_batch` RPC + longer tool descriptions
  on `fs_read` pointing the model at `fs_read_batch` and
  `code_outline`.
* **Compare against**: [v2](./2026-04-20-rust-v0.121.0-prefer-mcp.md),
  which already had `--prefer-mcp`.

## What actually happened

* **`fs_read_batch` only got called twice.** One of those calls did
  collapse 5 file reads into 1 batch — so the API works. But the
  agent didn't reach for it across the rest of its exploration; it
  still scroll-paged `mcp_connection_manager.rs` nine times in
  individual `fs_read` calls (vs six in v2 — *worse*).
* **The description nudge for `code_outline` got through:** 16 calls
  vs 0 in v2. But those calls added to the workload instead of
  replacing `fs_read`, so total MCP calls went up (144 vs 124).
* **Input tokens doubled** (2.45 M → 5.01 M) for three reasons: the
  new prose in `fs_read`'s description sits in context every turn;
  `code_outline` responses are structured JSON that costs its own
  tokens; and codex made ~20 more tool turns overall so the context
  replay grew each round.
* **Wall-clock doubled** directly tracks the extra turns — per-call
  daemon latency is still fast, but each additional MCP call pays a
  full model-reasoning round-trip.

## Takeaway

**Prose-only behaviour nudges in tool descriptions don't pay.** The
added text is a per-turn tax in input tokens; the model doesn't
reliably change strategy because we asked it to.

Two paths forward this result rules out or demotes:

1. ~~Longer descriptions that explain when to prefer a composed tool~~
   — negative ROI in this run.
2. ~~Ask the model to use `code_outline` before `fs_read`~~ — model
   does it once or twice, doesn't replace the scroll-page pattern.

And two that remain promising:

* **Revert the description puffery**, keep `fs_read_batch` as a
  minimal-prose opt-in. Measure the clean delta from the API alone.
* **Server-side reductions that don't require the model to change
  behaviour** — most concretely, a `search_grep ?context` mode that
  returns surrounding lines so the agent doesn't need a follow-up
  `fs_read` at all. 70 `search_grep` calls in this run; even partial
  adoption would pay back the `fs_read` queue behind them.
