# Codex fork/exec benchmark — sandbox ablation, `openai/codex@rust-v0.122.0`

* **Date**: 2026-04-21
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Target tree**: `openai/codex` at tag `rust-v0.122.0` (latest release)
* **Host**: macOS Darwin (arm64), single-sample N=1 per cell
* **Tracer**: `shim` (PATH-shadow wrappers, `$ZDOTDIR` override, `--add-dir` whitelist)
* **mcp-cli**: this repo at `main` post-`feat/bench-warm-cache`, includes
  the cold / warm two-pass harness (`--daemon-arg=--idle-timeout=30m`
  pinned into the generated codex config so the daemon survives
  between cold and warm passes)
* **Prompt**: [`bench/codex-forkexec/prompt.md`](../prompt.md)
* **Compare against**: [2026-04-20 `--prefer-mcp`](./2026-04-20-rust-v0.121.0-prefer-mcp.md)
  (the prior headline — sandboxed, v0.121.0, no warm pass)

## Headline

Disabling codex's built-in sandbox (`--sandbox workspace-write` →
`--sandbox danger-full-access`) flips the wall-clock verdict for
mcp-cli. With the sandbox on, cold mcp-cli was **49 % slower** than
baseline. With the sandbox off, cold mcp-cli is **44 % faster** than
baseline — **−93 s** on a 209 s baseline run. Cached-input tokens
drop by 1.24 M (−68 %) in the same cell.

The prior measurement's wall-clock regression was dominated by
Seatbelt enter/exit cost on every shell command the baseline fires,
not by the agent+daemon cost. Once that noise is gone, the daemon's
real effect becomes legible.

## What the sandbox was costing

The Seatbelt / Landlock wrapper codex uses for shell commands
involves syscall interposition, profile compilation, and fork cost
on every `rg`/`grep`/`cat` it runs. It is asymmetric:

* **Baseline** fires ~60 sandboxed shell commands per run. Each one
  pays the sandbox entry fee; collectively they dominate wall-clock.
  But because every bash pipeline `rg | head -3 | sort` is one
  sandbox enter (not three), the cost amortises.
* **mcp-cli** runs fire ~22 sandboxed shell commands (most of
  codex's own `git` internals), but the daemon itself runs outside
  the sandbox — so mcp-cli should *avoid* this tax. What actually
  showed up in the sandboxed run: codex's atomic-MCP-turn penalty
  (~2.7× more tool calls than bash pipelines) pushed wall-clock
  *higher* than baseline's sandbox overhead, because each model
  round-trip is also a latency unit.

Remove the sandbox and the picture inverts: baseline's atomic-call
count is still ~60 but each one is now cheap, so baseline gets
faster (−slightly — model variance dominates here); mcp-cli's
atomic-MCP-turn cost is unchanged, but the daemon's structured
returns pay off directly as fewer input tokens and a much smaller
context that the model has to chew through per turn. The result is
a genuine wall-clock win.

## Cold → warm under no sandbox

Warm regressed vs cold this run (+42 s, 159 s → 117 s). Model
variance is the direct cause: the warm agent chose to issue
**50 `search_grep`** calls (vs 30 in cold) and **16 `code_outline`**
(vs 12 in cold). It explored deeper because the daemon was already
warm and each probe came back fast — latency-for-depth trade.

Even so, warm beats the baseline across the board:

* **−24 %** wall-clock vs baseline (159 s vs 209 s)
* **−28 %** input tokens vs baseline (1.38 M vs 1.93 M)
* **−32 %** cached input vs baseline (1.22 M vs 1.81 M)

And the cold-to-warm cache-payoff number we care about for the
prewarm / search-cache claim: **cached input −649 k cold→warm**
despite warm making *more* calls. That's the in-memory cache doing
exactly what it's supposed to do.

## What's actually proven

1. **Sandbox noise was concealing a real win.** With it off, cold
   mcp-cli is 44 % faster wall-clock and 64 % cheaper on input
   tokens than baseline on the same prompt.
2. **The warm-cache mechanism works.** Cold→warm cached input
   drops 649 k tokens even as the agent makes more calls —
   `search_cache` + `parse_cache` + prewarm are amortising real
   work.
3. **The remaining regression (warm vs cold wall-clock) is model
   variance, not daemon perf.** Fifty search_grep calls at
   single-digit-ms each aren't the bottleneck; the agent's
   choice to make them is.

## Caveats

* **N = 1 per cell.** Model choices are the dominant source of
  variance. Trust the *direction* and rough magnitude, not the
  third significant figure. A follow-up N ≥ 3 per cell would
  tighten the warm-vs-cold comparison; the sandbox ablation delta
  is large enough that model variance can't flip its sign.
* **`danger-full-access` is only safe here because the prompt
  forbids mutations** and the bench runs in a throwaway
  `$TMPDIR` clone. Don't copy this flag into any production
  codex automation.
* **`--sandbox workspace-write` numbers from today differ from
  the 2026-04-20 v0.121.0 run** (`baseline=258.76`, `cold=375.64`)
  because the target ref bumped (`rust-v0.121.0` → `rust-v0.122.0`)
  and the codex CLI patched. The intra-day comparison against our
  own `164 → 245 → 283` row is the apples-to-apples one.

## Reproduce

```sh
cargo build --release
PATH="$PWD/target/release:$PATH" \
    MCP_CLI="$PWD/target/release/mcp-cli" \
    OUT_DIR=/tmp/mcpcli-codex-bench \
    bench/codex-forkexec/run.sh
```

The `run.sh` committed on `main` now passes
`--sandbox danger-full-access`. To re-measure under the sandbox,
revert that one line (the commit that flipped it is small and
self-contained).
