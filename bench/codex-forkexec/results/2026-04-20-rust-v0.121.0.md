# Codex fork/exec benchmark — `openai/codex@rust-v0.121.0`

* **Date**: 2026-04-20
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Target tree**: `openai/codex` at tag `rust-v0.121.0` (latest stable)
* **Host**: macOS Darwin (arm64), single-sample N=1
* **Tracer**: `shim` (PATH-shadow wrappers; counter dir whitelisted via `--add-dir`; `$ZDOTDIR` override so `zsh -lc` sees the shim)
* **mcp-cli**: this repo at `m5/shim-tracer-and-real-run` (release build), MCP plugin registered via `mcp-cli install --target codex`
* **Prompt**: [`bench/codex-forkexec/prompt.md`](../prompt.md) — analyse the codex source for three perf wins, prefer the `mcp__mcp-cli__*` tools when available

## Headline finding

**Mounting `mcp-cli` as an MCP plugin in `codex exec` does not reduce
fork/exec on this workload.** Per-binary execve deltas (-5, +1, +0,
-2, -4) are well inside single-sample model variance, and the only
`mcp_tool_call` events fired are codex's own startup discovery
(`list_mcp_resources`, `list_mcp_resource_templates`). **Zero calls
landed on `mcp-cli/fs_read`, `mcp-cli/search_grep`, etc.** even
though the prompt explicitly nudges the model toward them.

The wall-clock saving (-50 s, -18 %) and token shift (+4 % input,
-11 % output) are also within model variance — codex sampled a
shorter exploration path on the second run; the daemon never
served a request.

## Why nothing landed on the daemon

Codex's built-in `Read` / `Grep` / `Bash` tools shadow the same
capabilities `mcp-cli` exposes via MCP, and codex's tool router has
no preference rule that demotes them in favour of an MCP server's
`fs_read` / `search_grep`. The model gets a bigger toolbox but no
incentive to switch — its training reflexes go for `Bash("rg ...")`
and never check what the MCP plugin provides.

This is the same gap rtk-style projects close with a Claude-Code
**hook** that rewrites `Bash` arguments transparently. mcp-cli is
MCP-only today, so on codex it sits idle.

## What would actually move the needle on codex

* A codex-side tool-routing rule (config or feature flag) that
  prefers MCP `fs_read`/`search_grep` over the built-in `Read`/`Grep`
  when an MCP server provides them. Out of mcp-cli's hands; lives
  in the codex repo.
* A codex hook (analogous to Claude Code's `Bash` hook) that
  intercepts the agent's shell invocations and rewrites
  `cat`/`rg`/`git status` to the MCP tool — same compress-side win
  rtk gets, in mcp-cli framing.
* Workload-side: a prompt that **explicitly forbids** `Bash` for
  the cat/grep/git operations and forces the agent through the MCP
  surface. Synthetic but would prove the daemon path works.

## What this benchmark *did* prove

* The bench harness is reproducible end-to-end: clone target,
  install mcp-cli, run codex twice, count fork/exec, parse codex
  JSON for tokens + MCP tool calls, tabulate.
* The shim tracer (the macOS-no-root path) works inside codex's
  Seatbelt sandbox once `--add-dir "$OUT_DIR"` whitelists the
  counter directory and `$ZDOTDIR` overrides put the shim ahead of
  the user's `/opt/homebrew/bin` in `PATH` even under `zsh -lc`.
* M5's `metrics.tool_latency` snapshot path runs (the daemon
  responded to `metrics_tool_latency` between the bench's run.sh
  invocations), but the dump file came back empty because codex
  closed its bridge / the daemon idle-exited before run.sh's
  follow-up request landed. Tracked as a known gap; doesn't
  change the headline.

## Reproduce

```sh
cargo build --release
PATH="$PWD/target/release:$PATH" \
    OUT_DIR=/tmp/mcpcli-codex-bench \
    bench/codex-forkexec/run.sh
```

Codex must be installed and authenticated (`auth.json` under
`$HOME/.codex/`). The bench bootstrap copies it into a per-run
`CODEX_HOME` so the real config is left untouched.
