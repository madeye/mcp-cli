# Codex fork/exec benchmark — `openai/codex@rust-v0.121.0`, trimmed descriptions

* **Date**: 2026-04-20 (fourth iteration)
* **Codex CLI**: `0.120.0` (`codex exec --json`)
* **Config**: same as v2/v3 — `mcp-cli install --target codex --prefer-mcp`.
* **Change vs v3**: reverted the prose nudges in tool descriptions
  back to the short-form from v2. The `fs_read_batch` API stays
  available, just without the "PREFER this" text in its description.
* **Compare against**:
  [v2](./2026-04-20-rust-v0.121.0-prefer-mcp.md) and
  [v3](./2026-04-20-rust-v0.121.0-batch-v1.md).

## Delta vs v3 (the "with nudged descriptions" run)

| metric | v3 (nudged) | v4 (trimmed) |
|---|---:|---:|
| mcp input tokens | 5,012,261 | 1,804,819 |
| mcp wall clock (s) | 488.32 | 404.12 |
| mcp MCP calls | 144 | 118 |
| `code_outline` | 16 | 10 |
| `fs_read_batch` | 2 | **0** |

Trimming the description prose recovered ~3 M input tokens and
~80 s of wall clock. The nudge text was a per-turn tax paid in
every turn's context replay.

## What this rules out

* **Prose nudges in tool descriptions**. Negative ROI on this
  workload: the added text cost more in input-token replay than
  the model ever bought back in tool-selection wins.
* **`fs_read_batch` as a self-serve opt-in**. 0 uses here. The API
  is fine (v3 demonstrated one successful 5-path batch under
  nudged descriptions), but the agent doesn't reach for it
  unsolicited with a short description.

## What this keeps on the table

* **`fs_read_batch` stays shipped**, minus the prose. Useful for
  any caller that *does* plan ahead with multiple paths
  (including future compound-tool work or programmatic clients).
* **Agent-independent server-side reductions.** Specifically
  `search_grep` with a `?context: u32` field: return N lines
  around each hit so the agent doesn't need a follow-up `fs_read`
  to see context. 68 `search_grep` calls happened in this run;
  even partial adoption retires a share of the 36 `fs_read`
  calls behind them. Doesn't depend on the model choosing the
  right composition.

## Recurring theme

Wall-clock regression is ~3× more atomic tool calls than the
bash-pipeline baseline, multiplied by model-reasoning latency per
turn. Per-call daemon latency is not the bottleneck. Anything that
collapses two calls into one on the server side (context in
`search_grep`, maybe `fs_stat_batch` for size/mtime discovery, …)
buys real wall-clock.
